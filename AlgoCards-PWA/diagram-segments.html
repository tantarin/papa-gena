#32 Пересекающиеся отрезки

Вход: [[6,8],[1,5],[4,7]] · k=1 · Выход: все три пересекаются

Диаграмма 1 — Отрезки на числовой оси

Что видно

Все три отрезка пересекаются попарно через зоны [4,5] и [6,7]. Жёлтые зоны — это и есть пересечения.

Почему все помечаются

[1,5] ∩ [4,7] = [4,5] ✓
[4,7] ∩ [6,8] = [6,7] ✓
Значит все три «пересекаются хотя бы с одним».

Диаграмма 2 — Шаг 1: сортировка по началу
Исходный порядок (segments[]):
[6, 8] idx=0
[1, 5] idx=1
[4, 7] idx=2
↓ сортируем sortedIndices по segments[i][0]
sortedIndices[] — индексы в порядке возрастания начала:
sortedIndices[0]
idx=1 → [1, 5]
sortedIndices[1]
idx=2 → [4, 7]
sortedIndices[2]
idx=0 → [6, 8]
Сортируем массив индексов, а не сами отрезки — чтобы в конце intersectsWithAny[i] соответствовал оригинальному segments[i]. После сортировки отрезки идут слева направо: начало первого ≤ начало второго ≤ ...
Диаграмма 3 — Пошаговый обход алгоритма
Диаграмма 4 — Почему достаточно отслеживать maxEnd

Главная идея

Если отрезки отсортированы по началу, то новый отрезок пересекается с уже просмотренными тогда и только тогда, когда его начало ≤ maxEnd — максимальному концу среди просмотренных.

Зачем maxEnd, а не просто предыдущий?

Предыдущий отрезок мог закончиться раньше. maxEnd — самый «длинный хвост» из всех просмотренных. Если новый начинается позже него — он гарантированно не пересекается ни с одним.
